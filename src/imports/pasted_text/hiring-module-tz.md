# Техническое задание

## Модуль HRM: **“Найм”**

## 1. Общая информация

### 1.1. Назначение модуля

Модуль **“Найм”** предназначен для автоматизации полного цикла подбора сотрудников в HRM-системе в режиме **одного окна**:

* создание и публикация вакансий;
* сбор откликов и резюме;
* ранжирование кандидатов;
* ведение воронки подбора;
* планирование интервью;
* согласование кандидатов;
* перевод кандидата в статус “нанят”;
* запуск автоматизированного онбординга;
* сопровождение испытательного срока;
* контроль прогресса через сводные отчеты.

### 1.2. Цель реализации

Сократить ручную работу HR и руководителей, обеспечить прозрачность подбора и адаптации, а также связать этапы **Recruitment → Hiring → Onboarding → Probation** внутри единой HRM-платформы.

### 1.3. Контекст продукта

Модуль разрабатывается как часть существующего HRM-портала. По стилю и UX он должен быть согласован с текущим интерфейсом системы:

* светлый фон;
* мягкие серо-белые поверхности;
* красный — основной акцентный цвет;
* скругленные карточки и кнопки;
* спокойная типографика;
* минималистичный корпоративный интерфейс;
* боковое левое меню;
* верхняя навигация;
* карточная структура блоков;
* простые и понятные таблицы, фильтры, статусы, счетчики.

---

# 2. Бизнес-цели

1. Дать HR возможность закрывать вакансии внутри одной системы.
2. Ускорить обработку откликов и резюме.
3. Упростить коммуникацию между HR, нанимающим менеджером и кандидатом.
4. Уменьшить количество ручных действий при онбординге.
5. Сделать испытательный срок управляемым и измеримым.
6. Предоставить отчетность по вакансиям, кандидатам, этапам и адаптации новичков.

# 3. Пользователи и роли

## 3.1. Роли

### HR-менеджер

Может:

* создавать и редактировать вакансии;
* публиковать вакансии;
* просматривать отклики;
* управлять воронкой кандидатов;
* назначать интервью;
* запускать онбординг;
* смотреть отчеты;
* настраивать шаблоны адаптации.

### Нанимающий руководитель

Может:

* просматривать кандидатов по своим вакансиям;
* оставлять комментарии;
* согласовывать / отклонять кандидатов;
* участвовать в интервью;
* отслеживать прогресс адаптации нанятого сотрудника.

### Интервьюер

Может:

* видеть назначенные интервью;
* оставлять фидбек;
* выставлять оценку кандидату.

### Администратор HRM

Может:

* управлять справочниками;
* правами доступа;
* шаблонами процессов;
* интеграциями;
* системными настройками.

### Новый сотрудник

Может:

* видеть личный onboarding-план;
* получать материалы;
* подтверждать выполнение задач;
* смотреть встречи, задания, чек-листы;
* отслеживать прогресс испытательного срока.

# 6. Пользовательские сценарии

## 6.1. Создание вакансии

HR создает вакансию, заполняет форму, сохраняет в черновик или отправляет на согласование, после чего публикует.

## 6.2. Сбор откликов

Отклики попадают в список кандидатов по вакансии. Для каждого кандидата создается карточка.

## 6.3. Ранжирование резюме

Система позволяет:

* сортировать кандидатов по релевантности;
* присваивать рейтинг;
* фильтровать по ключевым параметрам;
* помечать кандидатов тегами;
* быстро переводить между этапами.

## 6.4. Назначение собеседования

HR открывает карточку кандидата, выбирает слот, участников и тип интервью. Система создает событие и отправляет уведомления.

## 6.5. Принятие решения

После интервью система собирает фидбек. HR переводит кандидата в один из статусов: оффер / резерв / отказ / нанят.

## 6.6. Запуск онбординга

После статуса “Нанят” HR нажимает одну кнопку **“Запустить адаптацию”**. Система:

* создает onboarding-план;
* назначает задачи;
* отправляет материалы;
* планирует встречи;
* запускает контрольные точки испытательного срока.

## 6.7. Контроль испытательного срока

HR и руководитель видят:

* прогресс по задачам;
* прохождение обучения;
* результаты контрольных встреч;
* риски;
* итоговую оценку.

# 11. Бизнес-правила

1. Вакансию нельзя опубликовать без обязательных полей.
2. Кандидат должен быть привязан хотя бы к одной вакансии.
3. Интервью нельзя создать без даты, времени и участников.
4. Кандидата можно перевести в статус “Нанят” только после финального решения.
5. После перевода в “Нанят” система должна предложить запуск онбординга.
6. Онбординг можно запустить:

   * автоматически;
   * вручную одной кнопкой.
7. У каждого onboarding-плана должен быть шаблон или набор задач.
8. По окончании испытательного срока должно быть зафиксировано решение.
9. Все изменения статусов должны логироваться в истории.
10. Все ключевые действия должны сопровождаться уведомлениями.

# 16. Acceptance criteria

## 16.1. По вакансиям

* HR может создать, сохранить и опубликовать вакансию.
* Вакансия отображается в списке со статусом и счетчиками.

## 16.2. По кандидатам

* HR видит список кандидатов и воронку.
* Кандидат переводится между этапами.
* Резюме и рейтинг доступны в карточке.

## 16.3. По интервью

* Интервью назначается из карточки кандидата.
* Интервью отображается в календаре.
* По интервью можно сохранить фидбек.

## 16.4. По найму

* Кандидат может быть переведен в статус “Нанят”.
* После этого доступна кнопка “Запустить адаптацию”.

## 16.5. По онбордингу

* Система создает план адаптации по шаблону.
* Новичку назначаются задачи, материалы и встречи.
* HR и руководитель видят прогресс.

## 16.6. По испытательному сроку

* Для нового сотрудника создается контроль испытательного срока.
* В отчетах видны риски, просрочки и текущий статус.
